## 1. What is RAG and why does it matter?

LLMs are trained on large but inevitably limited datasets — they have no access to private or recent data. **Retrieval-Augmented Generation (RAG)** addresses this by letting the model dynamically pull in external sources at query time.

As context windows grow (thousands → millions of tokens), LLMs increasingly function like a kind of "operating system kernel" connected to external data. RAG is the paradigm that makes this practical.

### The three-stage RAG pipeline

| Stage | What happens | Tools |
|---|---|---|
| **Indexing** | Documents are chunked, embedded, and stored in a vector store | Chroma, OpenAI Embeddings |
| **Retrieval** | A user query is embedded and matched to the closest document chunks | Retriever, cosine similarity |
| **Generation** | The LLM receives the retrieved chunks + the query and produces a grounded answer | Prompt template, GPT-3.5 |

### Quick code sketch

```python
# 1. Index
text_splitter = RecursiveCharacterTextSplitter(chunk_size=1000)
chunks = text_splitter.split_documents(docs)
vectorstore = Chroma.from_documents(chunks, OpenAIEmbeddings())

# 2. Retrieve + Generate
retriever = vectorstore.as_retriever()
chain = ({"context": retriever, "question": RunnablePassthrough()}
         | prompt | llm | StrOutputParser())

chain.invoke("What is task decomposition?")
```

LangSmith can be used throughout for tracing — it exposes the retrieved documents, the constructed prompt, and the final answer, making debugging straightforward.

## 2. Indexing — turning documents into vectors

The goal of indexing is to load documents into a **retriever** — a system that, given a query, returns the most relevant chunks.

### Why vectors?

Text can't be compared arithmetically. Vectors can. Two approaches:

| Method | How it works | Characteristics |
|---|---|---|
| **Sparse vectors** | Word-frequency counts across a vocabulary | High-dimensional, mostly zeros, traditional (early Google) |
| **Dense embeddings** | Learned, fixed-length vectors capturing semantic meaning | Lower-dimensional, captures synonyms and concepts |

Dense embeddings are what modern RAG uses. OpenAI's `text-embedding-3-small` produces **1,536-dimensional** vectors.

### The indexing workflow

1. **Split** — break documents into chunks sized to fit the embedding model's token limit (512–8,000+ tokens depending on the model).
2. **Embed** — pass each chunk through an embedding model to produce a dense vector.
3. **Store** — save vectors in a vector store (e.g. Chroma) alongside a reference back to the original text.

At query time, the user's question is embedded with the *same* model, and **cosine similarity** is computed between the query vector and all stored document vectors.

## 3. Retrieval — finding the right chunks

Retrieval is a **nearest-neighbour search** in embedding space.

Think of it in 3D: each document chunk is a point in space, positioned by its meaning. Similar chunks cluster together. Drop the query into the same space and find the nearest points — those are the relevant chunks.

### Key parameter: `k`

`k` controls how many chunks are returned. `k=1` returns the single closest chunk; higher values bring back more context at the risk of noise.

### Full pipeline recap

```
Split → Embed → Index → [query time] Embed query → KNN search → Return top-k chunks
```

LangChain's modular design means embedding models, indexes, loaders, and splitters can all be swapped independently. LangSmith lets you inspect every retrieval result.

## 4. Generation — producing the answer

Retrieved document chunks are **"stuffed" into the LLM's context window** alongside the user's question via a **prompt template**.

### Prompt template anatomy

```
System: Use the following context to answer the question.

Context: {context}   ← retrieved chunks go here
Question: {question} ← user query goes here
```

### Two chain modes

**Manual chain** — pass `{context, question}` explicitly to understand what's happening at each step.

**Automated chain (LCEL)** — plug the retriever directly into the chain. Call `invoke(question)` and retrieval + generation happen in one step.

```python
# Automated RAG chain
rag_chain = (
    {"context": retriever, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)
rag_chain.invoke("What is task decomposition?")
```

This is a **basic RAG pipeline**. The remaining videos address its limitations.

## 5. Query Translation — Multi-Query

### The problem

A poorly worded query produces an embedding that may not sit close to the relevant documents in vector space — causing retrieval to miss them entirely.

### The fix

Generate **multiple rewritten versions** of the original question from different angles, retrieve on each independently, then take the **union** of all retrieved documents.

```
Original query q
    ↓ (LLM rewrites)
q1, q2, q3, q4
    ↓ (retrieve on each)
D1, D2, D3, D4
    ↓ (union)
D = D1 ∪ D2 ∪ D3 ∪ D4
    ↓
Final RAG prompt → answer
```

This "shotgun" strategy increases the likelihood that at least one reformulation aligns well with the relevant documents. The output is parsed as a newline-delimited list.

## 6. Query Translation — RAG Fusion

RAG Fusion extends multi-query by adding one critical step: **reciprocal rank fusion (RRF)**.

### Multi-Query vs RAG Fusion

| Feature | Multi-Query | RAG Fusion |
|---|---|---|
| Query generation | Multiple rewrites | Same |
| Retrieval | Independent per query | Same |
| Aggregation | Flat union (no ranking) | **RRF → single ranked list** |
| Output | Unordered combined set | **Consolidated ranked list** |

RRF merges the separate ranked retrieval lists into one, balancing each document's position across all queries. The demo produced 4 queries → 4 retrieval lists → 6 unique ranked documents.

### When RRF shines

- Querying multiple vector stores simultaneously
- Large-scale retrieval where you want a clean top-k from a combined pool
- Any scenario where different phrasings surface complementary documents

## 7. Query Translation — Decomposition

For complex questions, a single retrieval pass is often insufficient. Decomposition **splits the question into sub-questions** and answers them sequentially or in parallel.

### Example

> "What are the main components of an LM-powered autonomous agent system?"

Decomposed into:
1. What is LM technology?
2. How does it work?
3. What are the specific components?
4. How do the components interact?

### Two answering strategies

**Sequential (IRC — Interleaved Retrieval with Chain-of-Thought)**

Each sub-question's answer becomes context for the next retrieval step. The prompt at each step includes: current sub-question + all previous Q&A pairs + new retrieval results. Best when sub-questions depend on each other.

**Parallel**

Answer all sub-questions independently, then concatenate. Simpler but less context-aware. Best when sub-questions are independent.

## 8. Query Translation — Step Back Prompting

Instead of decomposing into smaller parts, step back prompting does the opposite — it generates a **more abstract, generalised version** of the question.

### Example

| | |
|---|---|
| Original | "What is task composition for LLM agents?" |
| Step back | "What is the process of task composition?" |

### Workflow

1. **Generate step back question** — few-shot prompt asks the LLM to paraphrase into a more generic form.
2. **Two independent retrievals** — one on the original question, one on the step back question.
3. **Combine contexts** — both retrieval results are merged into the final prompt.
4. **Generate** — LLM produces a richer answer from the combined context.

### When it's most useful

Domains with **layered knowledge** — textbooks, technical manuals — where abstract concepts live in different sections from specific details. Few-shot example quality heavily influences output quality.

## 9. Query Translation — HYDE

HYDE (Hypothetical Document Embedding) addresses the semantic gap between short queries and long documents.

### The problem

```
User query (short, question-shaped)  →  embeds FAR from documents
Indexed documents (long, answer-shaped) →  live elsewhere in the space
```

### The fix

Instead of embedding the raw question, **generate a hypothetical answer passage** and embed that instead.

```
Question → LLM ("Write a paper passage to answer this") → Hypothetical doc
                                                                ↓
                                                          Embed hypothetical doc
                                                                ↓
                                                     KNN search → real docs
                                                                ↓
                                               real docs + original question → answer
```

The hypothetical document doesn't need to be accurate — it just needs to be **shaped like an answer**, so its embedding lands near real answer documents in vector space.

### Trade-offs

| | |
|---|---|
| Advantages | No index changes needed; easy to integrate; prompt-tunable for any domain |
| Considerations | Extra LLM generation step (latency + cost); effectiveness varies by domain |

## 10. Routing

After query translation, the query needs to go to the **right data source or prompt chain**. Routing is the decision layer.

### Logical routing

Uses LLM reasoning + function calling to classify the query and return a structured output (validated via Pydantic).

```python
# Schema constrains the LLM output
class RouteQuery(BaseModel):
    datasource: Literal["python_docs", "js_docs", "golang_docs"]

structured_llm = llm.with_structured_output(RouteQuery)
```

Example: Python question → Python docs retriever; JS question → JS docs retriever.

### Semantic routing

Embeds candidate prompts ahead of time. At query time, embeds the question and routes to the highest-similarity prompt via a runnable Lambda.

Example: "What is a black hole?" → higher similarity to Physics prompt → routed to physics chain.

### When to use which

| | Logical | Semantic |
|---|---|---|
| Source categories | Well-defined | Fuzzy |
| Output guarantee | Strict (schema) | Probabilistic |
| Implementation | Function calling + Pydantic | Embedding similarity |

## 11. Query Construction

Users write natural language; vector stores support **structured metadata filters**. Query construction bridges the gap.

### Example

```
Natural:    "Find me videos on Chat LangChain published after 2024"
Structured: { semantic: "Chat LangChain", filter: publish_date > 2024-01-01 }
```

### Workflow

1. Identify available metadata fields (dates, lengths, view counts, etc.)
2. Define a search schema covering semantic + structured filter types
3. Bind the schema to the LLM via function calling — forces schema-valid output
4. Pass the natural language query → LLM outputs a structured query object
5. Validate with Pydantic → run filtered semantic search

Function calling is what makes this reliable: the LLM is constrained to the schema and cannot invent field names or invalid filters.

## 12. Advanced Indexing — Multi-Representation

Standard indexing embeds raw chunks — noisy and not always retrieval-optimal. Multi-representation indexing **decouples the retrieval unit from the generation unit**.

### Two-store architecture

```
[Vector store]      ← stores LLM-generated summaries (for retrieval)
      ↕ document ID
[Document store]    ← stores full original documents (for generation)
```

### How it works

**Indexing:**
1. Load documents
2. Summarise each with an LLM — capturing key ideas and keywords
3. Embed summaries → vector store (with document ID)
4. Store full docs → document store (keyed by same ID)

**Retrieval:**
1. Query vector store → best-matching summary + document ID
2. Fetch full original doc from document store
3. Pass full doc to LLM for generation — no chunking needed

Summaries are lower-noise than raw chunks. Long-context LLMs can consume the whole original document, so you don't lose information from chunking. Inspired by the *proposition indexing* paper.

## 13. Advanced Indexing — RAPTOR

Standard KNN with `k=3` is insufficient for **broad questions requiring synthesis across many documents**. RAPTOR fixes this by indexing at multiple levels of abstraction.

### The tree structure

```
Level 3 (top)    Cluster-of-cluster summaries — broadest abstraction
      ↑
Level 2          Summaries of document clusters
      ↑
Level 1 (leaf)   Raw document chunks — specific detail
```

All levels are combined into a single vector store for unified retrieval.

### Build process

1. Load raw documents → leaf nodes
2. Embed all leaf chunks
3. Cluster similar chunks
4. LLM summarises each cluster → new higher-level node
5. Recurse: cluster the summaries → summarise → repeat (3 levels in the demo)
6. Store everything (raw chunks + all summaries) in one vector store

### Retrieval advantage

- Specific question → KNN hits leaf nodes (detailed chunks)
- Broad question → KNN hits higher-level summary nodes (consolidated context)

Modern LLMs with 100K–1M token context windows make RAPTOR even more powerful — large clusters can be summarised without information loss. Models used: Claude and OpenAI.

## 14. Advanced Indexing — ColBERT

Standard embedding models compress an entire document into **one vector**. For long or nuanced documents, this lossy compression degrades retrieval. ColBERT keeps **one embedding per token**.

### Scoring algorithm

```
For each query token q:
    compute similarity with every document token d
    take the max similarity across all document tokens

Final score = Σ max_sim(q) for all query tokens
```

This "late interaction" approach preserves fine-grained semantic information by comparing token pairs rather than document-level averages.

### Standard vs ColBERT

| | Standard | ColBERT |
|---|---|---|
| Document representation | Single vector | One vector per token |
| Query representation | Single vector | One vector per token |
| Similarity | One dot product | Max similarity per query token, then sum |
| Speed | Fast | Slower (token × token comparisons) |
| Accuracy | Good | Strong, especially on long/nuanced docs |

### Implementation with Ragatouille

```python
from ragatouille import RAGPretrainedModel

RAG = RAGPretrainedModel.from_pretrained("colbert-ir/colbertv2.0")
RAG.index(collection=documents, index_name="my_index")
results = RAG.search(query="What inspired Miyazaki?", k=3)

# Plug into LangChain
retriever = RAG.as_langchain_retriever(k=3)
```

ColBERT integrates with LangChain's standard retriever interface. Best suited for longer documents where single-vector compression loses critical detail.
